Tenant Governance Settings

🔴 Critical: Environment creation open to all users

Current value: disableEnvironmentCreationByNonAdminUsers = false

Any licensed user can create production and sandbox environments. This leads to uncontrolled environment sprawl, increases licensing and capacity costs, and creates ungoverned shadow IT. An employee could spin up a production environment, connect it to sensitive data sources, and share it externally — all without admin visibility.

pac admin update-tenant-settings --setting-name "disableEnvironmentCreationByNonAdminUsers" --setting-value "true"

🔴 Critical: Trial environment creation open to all users

Current value: disableTrialEnvironmentCreationByNonAdminUsers = false

Trials auto-expire after 30 days. Users may build real workloads in trials, losing data when they expire. Trials also consume capacity and create unmonitored environments where sensitive data could be processed without governance oversight.

pac admin update-tenant-settings --setting-name "disableTrialEnvironmentCreationByNonAdminUsers" --setting-value "true"

🟡 High: Developer environment creation open to all users

Current value: disableDeveloperEnvironmentCreationByNonAdminUsers = false

Developer environments are personal but still consume capacity. Unrestricted creation leads to environment sprawl and makes it harder to maintain an accurate inventory. Currently there are already 7 developer environments in this tenant.

pac admin update-tenant-settings --setting-name "powerPlatform.governance.disableDeveloperEnvironmentCreationByNonAdminUsers" --setting-value "true"

DLP Policy Overview

PolicyScopeConfidentialGeneralBlockedCreated
Default All Environments 0 1,622 0 2026-04-13

🔴 Critical: All 1,622 connectors in General — no data separation

The single DLP policy classifies every connector as "General" (non-business). This means any connector can communicate with any other connector — there is effectively zero data loss prevention. A maker could create a flow that reads from SharePoint or Dataverse and sends the data to a personal Gmail, Dropbox, or an arbitrary HTTP endpoint.

What could happen: An employee (maliciously or accidentally) builds a flow: Dataverse → HTTP connector → external API. Sensitive customer data is exfiltrated without any alert or block. During a compliance audit, you cannot demonstrate that data flows are controlled.

Reclassify sensitive connectors (Dataverse, SharePoint, SQL Server, Office 365, etc.) into the Confidential group. This prevents them from being used alongside non-business connectors.

🔴 Critical: No connectors in the Blocked group

The HTTP connector, custom connectors, and other high-risk connectors are not blocked in any environment. The HTTP connector allows calling any external API, completely bypassing the intent of DLP policies. Custom connectors can wrap arbitrary REST APIs.

What could happen: A maker adds an HTTP action in a flow that posts Dataverse records to an external webhook. Since HTTP is unblocked and in the same group as Dataverse, no DLP rule prevents this. Data leaves the organization boundary with no audit trail in DLP.

Block the HTTP connector, custom connectors, and other high-risk connectors in all production, sandbox, and default environments. Only allow them in designated development environments with proper justification.

Environment Settings Analysis

Settings sampled from: Default environment, Team Demos (Production), Alex Johnson (Developer)

🔴 Critical: Auditing disabled in all sampled environments

Affected: Contoso Platform (Default), Team Demos (Production), Alex Johnson (Developer)

Auditing is disabled (isauditenabled = No) in every environment checked, including production. Without auditing, you have no record of who accessed, modified, or deleted data. This is required for SOC 2, ISO 27001, GDPR, and HIPAA compliance. If a security incident occurs, there is no forensic trail to investigate.

pac env update-settings --name isauditenabled --value true --environment "<environment-id>"

🟡 Warning: Session timeout disabled in all sampled environments

Affected: Contoso Platform (Default), Team Demos (Production), Alex Johnson (Developer)

Session timeout is disabled (sessiontimeoutenabled = No). Users who walk away from a shared device or forget to log out remain authenticated indefinitely. This is a session hijacking risk, especially in shared workspaces, kiosks, or when employees use public computers.

pac env update-settings --name sessiontimeoutenabled --value true --environment "<environment-id>"
pac env update-settings --name sessiontimeoutinmins --value 60 --environment "<environment-id>"
